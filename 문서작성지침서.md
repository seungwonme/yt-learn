# 구현 가이드 문서 작성 지침서

> AI 협업 기반 튜토리얼 문서를 작성하는 방법론 가이드

## 📌 이 지침서의 목적

이 문서는 **구현가이드.md**와 같은 형식의 **AI 협업 기반 튜토리얼 문서**를 작성하는 방법을 안내합니다.

**대상**:

- 오픈소스 프로젝트 문서 작성자
- 기술 튜토리얼 작성자
- AI를 활용한 문서 자동화를 원하는 사람

## 🧰 도구 소개 섹션 작성 가이드

> **핵심 원칙**: 독자가 "왜 이 도구를 쓰는지"와 "무엇을 할 수 있는지"를 이해해야 합니다.

### 도구 소개가 필요한 이유

- 독자가 처음 접하는 라이브러리/프레임워크일 수 있음
- 왜 이 도구를 선택했는지 맥락 제공
- 설치 명령어만 보고는 용도를 알 수 없음

### 도구 소개 템플릿

```markdown
## 🧰 사용하는 도구들

이 프로젝트에서 사용하는 주요 도구들을 간략히 소개합니다.

### [도구명]

**한 줄 설명**: [이 도구가 무엇인지 한 문장으로]

**왜 사용하나요?**

- [장점 1]
- [장점 2]

**예시**:
[간단한 사용 예시 - 코드 또는 설명]

**공식 문서**: [URL]
```

### 실제 예시

#### shadcn/ui

```markdown
### shadcn/ui

**한 줄 설명**: 복사해서 붙여넣기 방식의 React UI 컴포넌트 모음

**왜 사용하나요?**

- 디자인이 깔끔하고 현대적
- 필요한 컴포넌트만 선택해서 추가 가능
- Tailwind CSS와 완벽 호환
- 코드를 직접 소유하므로 커스터마이징 자유로움

**예시**:
버튼 하나 추가하려면 `pnpm dlx shadcn@latest add button` 실행

**공식 문서**: https://ui.shadcn.com
```

#### Next.js

```markdown
### Next.js

**한 줄 설명**: React 기반의 풀스택 웹 프레임워크

**왜 사용하나요?**

- 서버/클라이언트 코드를 한 프로젝트에서 관리
- 파일 기반 라우팅으로 페이지 구성 간편
- API 라우트로 백엔드 기능 구현 가능
- 이미지 최적화, SEO 등 기본 제공

**예시**:
`src/app/page.tsx` 파일이 자동으로 메인 페이지(`/`)가 됨

**공식 문서**: https://nextjs.org/docs
```

#### Zod

```markdown
### Zod

**한 줄 설명**: TypeScript를 위한 데이터 검증 라이브러리

**왜 사용하나요?**

- API로 들어오는 데이터가 올바른 형태인지 검사
- 잘못된 데이터가 들어오면 명확한 에러 메시지 제공
- TypeScript 타입을 자동으로 생성

**예시**:
"검색어는 필수입니다" 같은 에러 메시지를 자동으로 처리

**공식 문서**: https://zod.dev
```

#### react-youtube

```markdown
### react-youtube

**한 줄 설명**: React에서 YouTube 영상을 쉽게 재생하는 컴포넌트

**왜 사용하나요?**

- YouTube 플레이어를 React 컴포넌트로 간편하게 사용
- 재생/일시정지/시간 이동 등 제어 가능
- 이벤트 핸들링 (재생 시작, 종료 등) 지원

**예시**:
`<YouTube videoId="영상ID" />` 한 줄로 영상 플레이어 추가

**공식 문서**: https://www.npmjs.com/package/react-youtube
```

### 도구 소개 작성 원칙

1. **전문 용어 피하기**

   - ❌ "SSR/SSG 지원 프레임워크"
   - ✅ "서버에서 미리 페이지를 준비해서 빠르게 보여주는 도구"

2. **비유 활용하기**

   - ❌ "컴포넌트 라이브러리"
   - ✅ "레고 블록처럼 조합할 수 있는 UI 부품 모음"

3. **실용적 예시 제공**
   - 설치 명령어
   - 간단한 사용 코드
   - "이걸 쓰면 이런 결과가 나온다" 설명

## 📦 환경 설정 vs AI 구현 구분

### 핵심 원칙

| 구분     | 환경 설정                       | AI 구현                   |
| -------- | ------------------------------- | ------------------------- |
| **정의** | 도구 설치, 폴더 생성, 설정 파일 | 실제 코드 작성, 기능 구현 |
| **방식** | 터미널에서 직접 실행            | AI에게 프롬프트로 요청    |
| **이유** | 정확한 명령어가 중요            | 맥락과 요구사항이 중요    |

### 환경 설정 (직접 실행)

**포함되는 작업**:

- 패키지 설치 (`pnpm add`, `pnpm dlx`)
- 폴더 구조 생성 (`mkdir`)
- 설정 파일 초기화 (`init`)
- 환경 변수 설정 (`.env` 파일)
- 개발 서버 실행 (`pnpm dev`)

**작성 형식**:

````markdown
### 1-1. [작업명]

터미널에서 실행:

```bash
[명령어]
```

**결과**: [이 명령어가 하는 일 설명]
````

**실제 예시**:

````markdown
### 1-1. UI 라이브러리 설치

터미널에서 실행:

```bash
pnpm dlx shadcn@latest init -d
```

**결과**: 예쁜 UI 컴포넌트들을 사용할 수 있는 환경이 준비됩니다.
````

````markdown
### 1-4. 환경 변수 파일 만들기

프로젝트 폴더에 `.env.example` 파일을 만들고 아래 내용을 복사:

```bash
# YouTube Data API v3 키
# 발급: https://console.cloud.google.com/apis/credentials
YOUTUBE_API_KEY=

# Google Gemini API 키
# 발급: https://makersuite.google.com/app/apikey
GOOGLE_GEMINI_API_KEY=
```

그 다음 `.env.local` 파일을 만들고 실제 API 키를 입력합니다.

**결과**: 환경 변수 설정 완료
````

**환경 변수가 환경 설정인 이유**:

- 내용이 정해져 있음 (API 키 이름, 형식)
- AI가 생성할 필요 없이 복사-붙여넣기로 충분
- 잘못 생성되면 프로젝트 전체가 동작하지 않음

### AI 구현 (프롬프트로 요청)

**포함되는 작업**:

- 컴포넌트 만들기
- API 라우트 구현
- 비즈니스 로직 작성
- 스타일링
- 버그 수정

**작성 형식**:

````markdown
### 1-2. [기능명]

**AI에게 이렇게 말하세요:**

```
[프롬프트 내용]
```

**원하는 결과**:

- [기대 결과 1]
- [기대 결과 2]
````

## ✍️ 범용적 AI 프롬프트 작성 가이드

> **핵심 원칙**: 특정 프로젝트 구조를 몰라도 작성할 수 있어야 합니다.

### 나쁜 프롬프트 vs 좋은 프롬프트

#### ❌ 프로젝트 종속적 (나쁨)

```
@src/components/search/SearchBar.tsx 파일을 만들어줘.
shadcn/ui의 Input 컴포넌트를 import해서 사용해.
onSearch prop을 받아서 검색어를 전달해.
```

**문제점**:

- 파일 경로를 알아야 함
- import 방식을 알아야 함
- props 설계를 이해해야 함

#### ✅ 범용적 (좋음)

```
검색창을 만들어줘.

기능:
- 검색어를 입력할 수 있는 입력창
- 검색 버튼 (돋보기 아이콘)
- 검색 버튼 클릭 시 검색 실행
- 로딩 중일 때 버튼 비활성화

스타일:
- shadcn/ui 컴포넌트 사용
```

**장점**:

- 원하는 "결과"를 설명
- 파일 경로나 코드 구조 몰라도 됨
- AI가 적절한 위치와 구조 결정

### 프롬프트 작성 4단계 공식

#### 1단계: 무엇을 만들지 (What)

```
[만들려는 것]을 만들어줘.
```

예시:

- "검색창을 만들어줘"
- "영상 플레이어를 만들어줘"
- "로그인 페이지를 만들어줘"

#### 2단계: 어떤 기능이 필요한지 (Features)

```
기능:
- [사용자가 할 수 있는 것 1]
- [사용자가 할 수 있는 것 2]
- [사용자가 할 수 있는 것 3]
```

예시:

```
기능:
- 검색어 입력
- 검색 버튼 클릭으로 검색 실행
- 엔터키로도 검색 가능
- 검색 중 로딩 표시
```

#### 3단계: 어떻게 보여야 하는지 (Appearance)

```
화면:
- [화면 요소 1]
- [화면 요소 2]

또는

상태별 화면:
- 로딩 중: [표시할 내용]
- 에러 발생: [표시할 내용]
- 정상: [표시할 내용]
```

예시:

```
상태별 화면:
- 검색 전: "YouTube 영상을 검색해보세요" 메시지
- 로딩 중: 스켈레톤 UI
- 결과 없음: "검색 결과가 없습니다"
- 결과 있음: 카드 형태로 영상 목록 표시
```

#### 4단계: 제약 사항 (Constraints)

```
사용할 것:
- [라이브러리/도구]

주의할 점:
- [버전 이슈나 특이사항]
```

예시:

```
사용할 것:
- shadcn/ui의 Card, Button 컴포넌트
- react-icons의 돋보기 아이콘

주의할 점:
- 모바일에서도 잘 보이게 반응형으로
```

### 완성된 프롬프트 예시

#### 기본형

```
검색창을 만들어줘.

기능:
- 검색어 입력
- 검색 버튼 클릭 시 검색 실행
- 로딩 중 버튼 비활성화

사용할 것:
- shadcn/ui 컴포넌트
```

#### 상세형

```
영상 카드를 만들어줘.

기능:
- 영상 썸네일, 제목, 채널명, 설명 표시
- 클릭하면 영상 상세 페이지로 이동
- 마우스 올리면 살짝 확대되는 효과

화면:
- 카드 형태
- 썸네일은 위쪽, 정보는 아래쪽
- 제목은 2줄까지만 표시 (넘치면 ... 처리)

사용할 것:
- shadcn/ui Card 컴포넌트
- Next.js Image (이미지 최적화)
```

#### API 구현형

```
YouTube 검색 API를 만들어줘.

기능:
- 검색어를 받아서 YouTube에서 영상 검색
- 검색 결과를 JSON으로 반환

입력:
- 검색어 (필수)
- 최대 결과 수 (선택, 기본값 10)

출력:
- 영상 목록 (제목, 썸네일, 채널명, 영상ID)
- 다음 페이지 토큰 (더보기 기능용)

사용할 것:
- YouTube Data API v3
- zod로 입력 검증
```

### 프롬프트 체크리스트

작성한 프롬프트가 아래 기준을 충족하는지 확인하세요:

- [ ] 파일 경로를 명시하지 않았다
- [ ] import 문을 지정하지 않았다
- [ ] 변수명이나 함수명을 강제하지 않았다
- [ ] "무엇을 만들지"가 명확하다
- [ ] "어떤 기능이 필요한지" 나열했다
- [ ] 개발 지식 없이도 이해 가능하다
- [ ] 다른 프로젝트에서도 비슷하게 사용 가능하다

### 피해야 할 표현

| 피해야 할 표현              | 대체 표현                    |
| --------------------------- | ---------------------------- |
| `@src/components/...`       | (경로 지정 안 함)            |
| `import { Button } from...` | "Button 컴포넌트 사용"       |
| `useState로 상태 관리`      | "검색어를 기억해둘 수 있게"  |
| `async/await 패턴 사용`     | "API 응답을 기다렸다가 처리" |
| `map으로 리스트 렌더링`     | "목록을 화면에 표시"         |
| `onClick 핸들러 추가`       | "클릭하면 ~하게"             |

## 🎨 섹션별 작성 가이드

### 1. 제목 및 소개

```markdown
# [프로젝트명] 만들기

> [한 줄 설명: 누가, 무엇을, 어떻게 만드는지]
```

**작성 팁**:

- 제목은 명확하고 간결하게
- 한 줄 설명은 대상 독자와 최종 결과물을 명시
- 예: "AI와 함께 코딩 없이 만드는 YouTube 학습 플랫폼"

### 2. 시작하기 전에

```markdown
## 시작하기 전에

이 가이드는 **[대상 독자]** 와 함께 [프로젝트]를 만드는 방법입니다.
[지식 수준 설명].

### 필요한 것

1. **[도구 1]** - [설명]
2. **[도구 2]** - [설명]
3. **[키/계정]** - 나중에 발급받을 예정
```

**작성 팁**:

- 대상 독자를 명확히 (비개발자, 초보, 중급 등)
- 필요한 사전 지식 수준 명시
- 필수 도구는 3-5개 이내로 제한

### 3. 주의사항 표시

````markdown
### [섹션] ⚠️ 중요 업데이트

**AI에게 이렇게 말하세요:**

```
[프롬프트]

⚠️ 중요:
- [주의사항 1]
- [주의사항 2]
```
````

**작성 팁**:

- ⚠️ 이모지로 중요 정보 강조
- 버전별 차이점 명시
- 구체적인 상황 설명

### 4. 새 기능 표시

## 📎 @ 기호 사용법

> **핵심 원칙**: `@` 기호는 프롬프트 내에서 **파일 경로**를 참조할 때만 사용합니다.

### @ 기호란?

AI 코딩 도구(Claude Code, Cursor 등)에서 `@` 기호는 **"이 파일을 읽고 참고해"** 라는 의미입니다.

- `@파일경로` → "이 파일을 읽고 참고해" ✅
- `URL` → 그냥 일반 링크로 작성 (@ 없이) ✅

### 왜 공식 문서 참조가 중요한가?

**AI는 최신 정보를 모를 수 있습니다.**

- AI의 학습 데이터에는 시점 제한이 있음
- 라이브러리가 업데이트되면 사용법이 바뀔 수 있음
- 공식 문서를 참조하면 AI가 **최신 정보를 기반으로** 코드를 작성

**예시: 버전에 따른 차이**

```
❌ 공식 문서 없이 요청:
"Zod로 입력 검증해줘"
→ AI가 옛날 버전 문법을 사용할 수 있음 (error.errors vs error.issues)

✅ 공식 문서와 함께 요청:
"Zod로 입력 검증해줘.

참고: https://zod.dev/ERROR_HANDLING"
→ AI가 공식 문서를 읽고 최신 문법 사용
```

**프롬프트 설계 원칙**:

1. **구현에 필요한 공식 문서를 항상 포함**
   - API 사용법 → 해당 API 공식 문서
   - 라이브러리 사용 → npm/공식 사이트 문서
   - 프레임워크 기능 → 프레임워크 공식 문서

2. **구체적인 페이지를 참조**
   - ❌ `https://nextjs.org` (너무 광범위)
   - ✅ `https://nextjs.org/docs/app/api-reference/functions/fetch` (구체적)

3. **문서 작성자가 미리 찾아서 제공**
   - 독자가 문서를 찾을 필요 없이
   - 프롬프트에 필요한 공식 문서 링크를 포함해서 작성

**좋은 프롬프트 구조**:

```
[만들려는 것] 만들어줘.

기능:
- [기능 1]
- [기능 2]

참고: [공식 문서 URL]
```

### 사용 예시

#### 파일 참조 (수정 요청 시)

```
검색 결과 컴포넌트를 수정해줘.

@src/components/search/SearchResults.tsx

로딩 중일 때 스켈레톤 UI를 표시하도록 변경해줘.
```

#### 공식 문서 참조 (URL은 @ 없이)

```
YouTube 자막 추출 기능을 만들어줘.

기능:
- 영상 ID로 자막 가져오기
- 타임스탬프와 텍스트 추출

참고: https://github.com/LuanRT/YouTube.js
```

#### 파일과 URL 함께 사용

```
AI 요약 API를 만들어줘.

기능:
- 자막을 받아서 요약 생성
- JSON 형식으로 응답

@src/types/index.ts (타입 정의 참고)
참고: https://ai.google.dev/gemini-api/docs
```

### @ vs 일반 링크 구분

| 상황 | 사용 방법 | 예시 |
|------|----------|------|
| 파일 참조 | `@파일경로` | `@src/types/index.ts` |
| URL 참조 | `참고: URL` | `참고: https://zod.dev` |
| 패키지명 | 그대로 사용 | `pnpm add @google/generative-ai` |

### 주의사항

1. **@ 기호는 파일 경로에만**
   - ✅ `@src/components/SearchBar.tsx`
   - ❌ `@https://nextjs.org`
   - ✅ `참고: https://nextjs.org`

2. **URL은 "참고:" 형식으로**
   - 프롬프트 안: `참고: https://...`
   - 본문: `참고: https://...`

3. **npm 패키지명은 예외**
   - `@google/generative-ai`, `@types/node` 등은 패키지명이므로 그대로 사용

4. **너무 많은 참조는 피하기**
   - 파일 참조 + URL 참조 합쳐서 1-3개 정도가 적당
   - 너무 많으면 AI가 혼란스러워할 수 있음

## 🎓 마무리

**효과적인 구현 가이드의 3가지 원칙**:

1. **도구를 이해시켜라** - 왜 이 도구를 쓰는지 설명
2. **환경 설정은 직접, 구현은 AI에게** - 역할 구분 명확히
3. **프롬프트는 범용적으로** - 특정 프로젝트에 종속되지 않게

**좋은 프롬프트의 기준**:

- 개발자가 아닌 사람도 쓸 수 있다
- 다른 프로젝트에서도 비슷하게 적용된다
- "무엇을 원하는지"가 명확하다
- 파일 경로나 코드 구조를 강제하지 않는다
